4. Fluxo de Negócio
Fluxo Principal
O usuário acessa a tela Faturamento On Demand.
1.
O usuário seleciona a visão Billnow.
2.
O sistema identifica a necessidade de consulta de status de faturamento.
3.
A tela chama a API de listagem de faturas prevista na
US PTI7955
-
352
com filtros,
identificadores e correlationId.
4.
A API valida payload, contexto e perfil de acesso.
5.
A API chama o
OPCODE C_OP_LIST_INVOICE.
6.
O OPCODE consulta os dados da fatura e os campos de controle do Billnow na origem
definida.
7.
A API retorna status, datas, identificadores e dados resumidos de invoice.
8.
A tela atualiza a grid de resultados com o status retornado.
9.
Fluxos Alternativos
FA01
: Quando o faturamento estiver em processamento, o sistema deve retornar
BILLNOW_PROCESSING e a tela deve exibir Em execução.
•
FA02
: Quando
FLG_ENV_CARGA_CYBER = 1 e FLG_ENV_CARGA_HUB = 1
, o
sistema deve retornar BILLED e a tela deve exibir Billnow finalizado.
•
Fluxos de Exceção
FE01
: Quando não houver identificador ou filtro mínimo conforme contrato da US
PTI7955
-
352, a consulta não deve ser iniciada ou deve retornar erro funcional controlado.
•
FE02
: Quando o registro não for encontrado, o sistema deve retornar resposta funcional
padronizada conforme contrato da listagem.
•
FE03
: Quando houver divergência entre o identificador do SFA e o registro a ser
processado, o sistema deve retornar erro de correlação para tratamento manual.
•
FE04
: Quando houver falha técnica, o sistema deve retornar erro padronizado sem expor
detalhes internos.
•
FE05
: Quando o usuário não estiver autorizado, o usuário não deve ter acesso à tela ou à
funcionalidade.
